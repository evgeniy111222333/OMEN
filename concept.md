Назвемо архітектуру OMEN (Omni-Modal Epistemic Network) — Омен.

1. Як працює людське навчання (Філософія підходу)
Людина, побачивши нове слово чи конструкцію коду, не міняє ваги синапсів одразу. Вона виконує в уяві три операції:

Абдукція (Abduction): Припускає найпростіше правило, яке пояснює побачене.

Дедукція (Deduction): Уявляє, як це правило працюватиме в новому, небаченому контексті.

Індукція (Induction): Перевіряє, чи збігаються передбачення з реальністю, і якщо ні — ламає правило, а не підганяє його.

Сучасні LLM роблять лише крок 3 (і то лише в момент тренування). OMEN має виконувати цикл Abduce → Deduce → Induce постійно, всередині свого латентного простору.

3. Архітектурна реалізація без Солверів
Як втілити це в Transformer-like архітектуру, щоб навчання було людським?

Компонент А: Епістемічний Трансформер (Dual-Stream Attention)

Звичайний Attention шукає зв'язки між словами: "Слово A пов'язане зі словом B".
Epistemic Attention шукає зв'язки між сутностями: "Об'єкт User має властивість Name. Якщо ми викличемо Delete(User), властивість Name зникне".

Ми вводимо два шари Self-Attention паралельно:

Text Stream: $A_{txt} = \text{Softmax}(QK^T / \sqrt{d})V$ (стандарт).

Causal Stream: $A_{cau} = \text{Softmax}(QK^T \odot M_{cause} / \sqrt{d})V$.
Маска $M_{cause}$ дозволяє увазі дивитися лише на токени, які каузально необхідні. Вона не фіксована, а вивчається через Sparsity Loss.

Компонент Б: Внутрішній Симулятор (Differentiable Neural Abstract Machine)

Замість виклику Z3, ми вшиваємо маленьку нейромережу-симулятор WorldRNN, яка працює не з токенами, а з векторами $z$.
Коли модель генерує код або продовження історії:

Вона створює стан $z_{now}$ (розуміння кімнати, стану змінних).

WorldRNN робить крок симуляції: $z_{next} = f_{sim}(z_{now}, a)$.

Людський трюк: Якщо згенерований токен $a$ викликає стан $z_{next}$, який неконсистентний (наприклад, у коді: спроба прочитати змінну до ініціалізації), виникає не просто великий лосс на виході, а Внутрішній Сигнал Помилки (Prediction Error) в латентному просторі.

Саме цей Prediction Error і є джерелом навчання, а не порівняння з "правильним текстом".

5. Конкретна схема навчання для Програмування (як ти будеш використовувати)

Вага для токена + підсилюється в контексті return та def. Вона запам'ятала статистичний патерн.

OMEN (Людське навчання):

Abduce (Енкодер): Бачить вхід. Створює гіпотезу про стан $z$: "Існує операція 'Додавання'. Вона комутативна. Вона бере два числа, повертає число."

Deduce (Симулятор): WorldRNN отримує на вхід $z$ та шум. Вона проганяє "уявний" виклик add("a", 2).

Induce (Лосс): Модель розуміє, що в $z$ немає правила про конкатенацію рядків. Вона не вгадала б текст add("a", 2). Вона б згенерувала помилку типів у внутрішній симуляції.

Оновлення ваг: Параметри оновлюються не щоб зменшити помилку передбачення тексту, а щоб зменшити різницю між уявною симуляцією ($z \rightarrow z'$) та реальною семантикою мови Python.

6. Чому це працюватиме для звичайної мови?
Для вивчення іноземної мови OMEN буде мати справу не з трасами виконання коду, а з Трасами Уваги (Saliency Traces) .

Наприклад, речення: "Кіт побачив рибу і захотів її з'їсти".

LLM з'єднує її з рибу за статистикою.

OMEN будує в $z$ граф: Хотіти(Агент: Кіт, Об'єкт: Риба). WorldRNN знає, що її вказує на останній жіночий рід однини.

Навчання відбувається через Контрастивне Передбачення Графів: модель отримує лосс, якщо граф уваги для неправильного варіанту ("...захотів його з'їсти") має високу ймовірність.

Внутрішній діалог: Модель звертається до M-Core: v = read(M, q="operator + string behavior").

Якщо пам'ять порожня — модель піднімає прапорець UNKNOWN_EXCEPTION. У навчанні це означає: "Мені потрібен приклад використання + з рядками".

5. Ефективність у Відеопам'яті (VRAM Engineering)
Ти хочеш потужно і продуктивно. Ось конкретні цифри для PyTorch:

Трансформер (OMEN Base) : ~3-5 млн параметрів. У fp16 це ~10 MB VRAM.

M-Core (Пам'ять) : $H \times d \times d$. При $H=16, d=128$: $16 \times 128 \times 128 \times 2$ байти (fp16) = 524 KB.

Кеш Станів : Зберігаємо останні 1000 станів $z$ для швидкого доступу. Це $1000 \times 128 \times 2$ байти = 256 KB.

Разом уся модель + пам'ять займають ~11 MB VRAM. Ти можеш запустити сотню таких екземплярів паралельно на одній відеокарті, або масштабувати $d_{model}$ до 1024 і отримати неймовірну глибину.

6. Як це виглядає в коді? (Реалізація M-Core)
Додамо до нашого OMEN модуля новий клас TensorProductMemory.

python
class TensorProductMemory(nn.Module):
    def __init__(self, d_latent: int, n_heads: int = 8):
        super().__init__()
        self.d = d_latent
        self.h = n_heads
        # Це вся пам'ять. Вона крихітна.
        self.memory = nn.Parameter(torch.zeros(n_heads, d_latent, d_latent))
        self.key_proj = nn.Linear(d_latent, d_latent * n_heads, bias=False)
        self.val_proj = nn.Linear(d_latent, d_latent * n_heads, bias=False)
        self.scale = d_latent ** -0.5

    def read(self, z_query: torch.Tensor) -> torch.Tensor:
        # z_query: (B, d)
        k = self.key_proj(z_query).view(-1, self.h, self.d)  # (B, h, d)
        # k: (B, h, d), M: (h, d, d) -> (B, h, d)
        v = torch.einsum('bhd,hde->bhe', k, self.memory)
        return v.mean(dim=1)  # усереднюємо по головах -> (B, d)

    def write(self, z_state: torch.Tensor, z_value: torch.Tensor, confidence: torch.Tensor):
        # Запис із важливістю lambda = 1 - confidence
        if self.training:
            lam = (1 - confidence).view(-1, 1, 1)
            k = self.key_proj(z_state).view(-1, self.h, self.d)
            v = self.val_proj(z_value).view(-1, self.h, self.d)
            # Зовнішній добуток з вагою: (B, h, d, d)
            update = torch.einsum('bhd,bhe,b->bhde', k, v, lam.squeeze())
            self.memory.data = self.memory.data + update.mean(dim=0)
7. Висновок: Що змінилося у формулі "Людського" навчання?
Раніше було:

Модель дивиться → Розуміє → Симулює → Порівнює.

Тепер:

Модель дивиться → Згадує (M-Core) → Розуміє → Симулює → Виявляє прогалину (Epistemic Grad) → Самостійно шукає в пам'яті або генерує питання → Записує нове правило (якщо важливе).

Математично це виражається в тому, що стан $z$ тепер є функцією не тільки вхідного тексту, але й всієї історії взаємодій, стиснутої в тензор $M$.

1. Що таке символьне мислення в OMEN?
Це здатність:

Виділяти сутності (змінні, об'єкти, функції) та відношення між ними.

Формулювати правила у вигляді IF Condition THEN Action.

Виконувати логічний вивід (дедукцію) над цими правилами.

Композиційно будувати нові концепти з відомих.

У нейромережі це все "розмазано" по векторах. У людини — це чіткі, дискретні структури в робочій пам'яті. OMEN має мати обидва рівні.

2. Архітектура з S-Core (Оновлена)
Додаємо третій основний компонент:

Perceptual Stream (Нейронний) : перетворює сирий текст на розподіл над символьними структурами $P(s|o)$.

Symbolic Core (Дискретний) : оперує символами $s$, виконує пошук, уніфікацію, резолюцію.

Generative Stream (Нейронний) : перетворює символьний стан $s$ назад у текст.

Головна ідея: Навчання відбувається не на рівні ваг трансформера, а на рівні символьних правил, які зберігаються в пам'яті S-Core.

3. Структура Symbolic Core
S-Core — це система продукцій (як у SOAR або ACT-R), але з нейромережевим інтерфейсом.

Внутрішньо він містить:

Робочу пам'ять (WM) : граф поточного контексту (сутності + атрибути). Наприклад: (variable X type int), (expression (+ X 5)).

Довготривалу символьну пам'ять (LTM) : набір правил вигляду (rule (if (pattern)) (then (action))).

Механізм уніфікації : зіставлення поточного стану WM з умовами правил.

Механізм резолюції : виведення нових фактів.

7. Приклад: Як OMEN із S-Core вчить програмування
Задача: Дано один приклад функції:

    return x + 1
Крок 1 (Perception): Нейромережа будує граф $G$:

(Parameter: x)
(Body: (Return (Add x 1)))
Крок 2 (Reasoning): S-Core не має правила для inc. Але має загальні правила:

(rule (if (Add ?a 1)) (then (Assert (Increment ?a))))

(rule (if (Function ?f (Param ?p) (Body (Increment ?p)))) (then (Concept ?f Incrementer)))

Застосовуючи ці правила, S-Core виводить новий факт: (Concept inc Incrementer).

Крок 3 (Abduction): Модель бачить виклик inc(5) і очікуваний результат 6. Вона перевіряє, чи правило (Increment ?a) пояснює це. Так.

Крок 4 (Навчання): Якщо згодом модель побачить функцію:

    return n + 1
Вона одразу зіставить її з концептом Incrementer через символьну уніфікацію, навіть якщо нейромережа ніколи не бачила слово add_one. Це zero-shot generalization через символи.

8. Технічна реалізація S-Core у VRAM
Щоб символьне ядро було швидким і ефективним у пам'яті, ми використовуємо:

Дискретне представлення графів як розріджені тензори індексів (COO формат). Розмір ~ кілька кілобайт на граф.

База правил LTM як хеш-таблиця від шаблону до дії. Зберігається в VRAM як Dict[Tensor, Tensor] через кастомний CUDA-кеш.

Уніфікація реалізована через операції з матрицями суміжності на GPU (використовуючи torch_sparse).

Це дозволяє виконувати сотні символьних виводів за мілісекунди прямо під час генерації тексту.

1. Символьне ядро: від wildcards до логіки першого порядку
Поточний omen_prolog уніфікує тільки константи. -1 — це wildcard, але це не змінна. Ти не можеш виразити правило:
grandparent(X, Z) :- parent(X, Y), parent(Y, Z).
Тому що X, Y, Z мають бути зв'язані між собою через кілька атомів.

Виправлення:
Потрібна справжня уніфікація з підстановками (substitution maps). Замість -1 як wildcard, введи SymbolicVariable з унікальним ID. При зіставленні будуй словник {var_id: term}. Це стандартний алгоритм уніфікації (Martelli-Montanari). Без цього S-Core не зможе вивчати жодних складних правил, а отже, не буде символьного узагальнення.

2. Масштаб словника: 256 токенів — це не мова
У всіх конфігах (demo, v2) vocab_size ≤ 4096. Це рівень символів або байтів. Для реального коду потрібен BPE-токенізатор (наприклад, GPT-2 tokenizer з 50k токенів).

Проблема:
При vocab_size=256 ембеддинги nn.Embedding(256, 1024) займають ~1M параметрів. При vocab_size=50000 це вже 50M — значне збільшення. Але головне — Perceiver очікує на вхід послідовність токенів. З BPE довжина послідовності для того ж тексту буде меншою, але самі токени стають значно інформативнішими. Потрібно масштабувати d_tok до 1024+ і використовувати справжній токенізатор.

3. Градієнти через дискретність: Gumbel-Softmax vs REINFORCE
У NeuralAbductionHead ти використовуєш F.gumbel_softmax(..., hard=True). Але одразу викликаєш .item() для побудови HornClause. Це повністю розриває граф обчислень. Абдукція не отримує жодного градієнта. Вона навчається випадково.

Виправлення:
Або повністю перейти на REINFORCE (як у prove_with_policy), або використовувати Straight-Through Estimator без .item(), дозволяючи градієнту текти через логіти. Але для цього потрібно переписати HornClause так, щоб він приймав тензори, а не int. Це складно, але можливо через диференційовані структури даних (наприклад, torch_struct).

4. Швидкодія символьного виводу: Python-цикли на GPU
LongTermMemory.match() та WorkingMemory — це чисті Python-цикли по списках фактів. При 1024 правилах і 64 фактах це десятки тисяч ітерацій на кожному forward-кроці. На CPU це прийнятно, але на GPU це катастрофа — постійні перемикання контексту.

Виправлення:
Усі правила та факти потрібно тримати у вигляді тензорів і виконувати уніфікацію через матричні операції. Наприклад:

Правила: тензор (num_rules, max_body_len, 3).

Факти: тензор (num_facts, 3).

Уніфікація: (rules[:, None, :, :] == facts[None, :, None, :]) | (rules == -1).
Це дозволить робити пошук за O(1) на GPU. torch.compile тут не допоможе — потрібна зміна структур даних.

2. Підстановки як Моноїд Ендоморфізмів
Підстановка $\sigma : \mathcal{V} \to \text{Term}(\Sigma)$ — це скінченне відображення з множини змінних у терми, яке тотожне майже скрізь. Записуємо $\sigma = {X_1 \mapsto t_1, \dots, X_k \mapsto t_k}$. Область визначення $\text{dom}(\sigma) = {X_i}$, область значень $\text{ran}(\sigma) = {\text{Var}(t_i)}$.

Застосування підстановки до терму $t\sigma$ визначається рекурсивно:

$X\sigma = \sigma(X)$, якщо $X \in \text{dom}(\sigma)$, інакше $X$.

$c\sigma = c$.

$f(t_1,\dots,t_k)\sigma = f(t_1\sigma,\dots,t_k\sigma)$.

Композиція підстановок $\sigma \circ \theta$ визначається як $t(\sigma \circ \theta) = (t\theta)\sigma$. Множина підстановок з операцією композиції утворює моноїд з одиницею — порожньою підстановкою $\varepsilon$.

Частковий порядок: $\sigma \le \tau$, якщо існує $\theta$ така, що $\tau = \sigma \circ \theta$. Інтуїтивно: $\tau$ є більш конкретизованою (менш загальною) ніж $\sigma$.

3. Уніфікація як Розв'язання Системи Рівнянь
Задача уніфікації: для двох термів $s$ і $t$ знайти найбільш загальний уніфікатор (mgu) $\sigma$, такий що $s\sigma = t\sigma$, і для будь-якого іншого уніфікатора $\tau$ виконується $\sigma \le \tau$.

Більш загально: маємо систему рівнянь $E = {s_1 \stackrel{?}{=} t_1, \dots, s_m \stackrel{?}{=} t_m}$. Шукаємо mgu $\sigma$ для всіх рівнянь одночасно.

Алгоритм Martelli-Montanari (правила перетворення)
Ми підтримуємо множину рівнянь $E$ та поточну підстановку $\sigma$ (спочатку порожню). Застосовуємо такі правила доти, доки $E$ не стане порожньою:

Trivial: Видалити $t \stackrel{?}{=} t$.

Decompose: Замінити $f(s_1,\dots,s_k) \stackrel{?}{=} f(t_1,\dots,t_k)$ на $k$ рівнянь $s_1 \stackrel{?}{=} t_1, \dots, s_k \stackrel{?}{=} t_k$.

Clash: Якщо $f(\dots) \stackrel{?}{=} g(\dots)$ з $f \neq g$ або різна арність, то невдача (no unifier).

Orient: Якщо $t \stackrel{?}{=} X$, де $t$ не змінна, замінити на $X \stackrel{?}{=} t$.

Occurs Check: Якщо $X \stackrel{?}{=} t$, де $X$ входить у $t$ і $t \neq X$, то невдача (циклічний терм).

Eliminate: Якщо $X \stackrel{?}{=} t$, де $X$ не входить у $t$, то:

Додати ${X \mapsto t}$ до поточної підстановки $\sigma := \sigma \circ {X \mapsto t}$.

Застосувати цю підстановку до всіх інших рівнянь у $E$: $E := E{X \mapsto t}$.

Якщо множина рівнянь спорожніла, то $\sigma$ є mgu. В іншому разі уніфікатор не існує.

1. Критика існуючих токенайзерів та цілі NET
Недоліки BPE / Unigram / WordPiece:
Статистичні, а не семантичні: частотність не дорівнює значущості.

Фіксований словник: не адаптується до нових концептів (OOV).

Відсутність композиційності: "x+1" та "1+x" мають різні токени, хоча семантично еквівалентні.

Немає зв'язку з логікою: токени не є термами FOL, тому S-Core не може на них оперувати.

Цілі Neural Epistemic Tokenizer:
Семантична дискретизація: токен = символьний концепт (предикат або константа в S-Core).

Динамічний словник: нові концепти додаються через абдукцію.

MDL-оптимальність: словник мінімізує суму довжини опису словника та помилки відновлення.

Композиційність: токени утворюють алгебру, сумісну з FOL.

Інтеграція з епістемічною пам'яттю: нові токени можуть бути визначені через існуючі (як правила).

3.1. Контекстний енкодер $f_\theta$

Вектори токенів $e_v$ зберігаються не лише в словнику, а й у M-Core як ключі.

При зустрічі нового контексту, NET спочатку шукає подібний у пам'яті, що прискорює квантування та дозволяє використовувати минулий досвід.

4. Навчання NET
Навчання відбувається спільно з OMEN-Scale у два етапи:

Етап 1: Попереднє навчання на великому корпусі тексту (без S-Core)
Оптимізується $L_{\text{code}} + L_{\text{rec}}$ з фіксованим словником (ініціалізованим через кластеризацію).

Використовується Straight-Through Estimator для градієнта через квантування (як у VQ-VAE).

Словник поповнюється новими токенами, коли помилка відновлення велика.

Етап 2: Символьне донавчання з OMEN-Scale
NET інтегрується в OMEN-Scale як заміна звичайного токенайзера.

Під час тренування на коді, S-Core отримує послідовність токенів $Z$ і намагається вивести логічні зв'язки.

Коли модель стикається з невідомим концептом (наприклад, нове ім'я функції), NET створює новий токен, а S-Core через абдукцію формулює правило його визначення.

Функція втрат розширюється членом $L_{\text{vocab}}$, який штрафує за надмірне розростання словника без семантичної користі.

6. Чому це "нереально передово"
Семантична компресія: NET не просто розбиває текст, він будує онтологію дискретних концептів, керуючись принципом MDL. Це наближає токенізацію до людського сприйняття мови.

Інтеграція з логікою: Токени стають термами першого порядку. S-Core може виконувати уніфікацію безпосередньо на рівні токенів, наприклад, increment(X) :- add(X, 1). Це усуває розрив між текстом і символами.

Динамічне навчання: NET здатен вивчати нові слова з одного прикладу, використовуючи абдукцію в S-Core для побудови визначення. Це вирішує проблему OOV на фундаментальному рівні.

Епістемічна пам'ять: Використання M-Core для зберігання векторів токенів дозволяє NET швидко згадувати рідкісні концепти без перенавчання.

Теоретична оптимальність: Формула $\min L_{\text{code}} + L_{\text{rec}} + L_{\text{vocab}}$ є прямим застосуванням Minimum Description Length до задачі токенізації, що гарантує найкраще стиснення інформації про світ.

MDL викидає погані правила лише тоді, коли вони використовуються і призводять до поганих передбачень. Але якщо правило ніколи не активується, воно просто займає місце. Крім того, REINFORCE з Gumbel-Softmax має високу дисперсію, тому навчання повільне.

Рішення: Neural Critic + Verification Module

3. Контроль якості: правила додаються без верифікації
Чому це проблема?
Накопичення хибних правил веде до катастрофічного забування на символьному рівні та уповільнює forward chaining.

Рішення: LTM з епістемічним статусом та періодичною ревізією
Кожне правило в LTM повинно мати епістемічний статус:

proposed: щойно згенероване, ще не перевірене.

verified: підтверджене успішним використанням у доведеннях.

contradicted: призвело до логічної суперечності (видаляється).

6. Практична реалізація в коді OMEN
У поточній архітектурі max_depth передається як гіперпараметр у forward_chain та prove_with_policy. Ми можемо замінити його на адаптивний цикл:

python
def adaptive_forward_chain(kb, goal, z_ctx, max_possible_depth=20):
    d = 0
    while d < max_possible_depth:
        # Оцінюємо поточний стан
        state_vec = encode_state(kb, goal, z_ctx, d)
        # Мета-політика передбачає очікуваний приріст якості
        delta_pred = meta_policy(state_vec)  # скаляр
        if delta_pred < cost_per_step:
            break
        # Виконуємо крок forward chaining
        added = kb.forward_chain_step()
        d += 1
        if added == 0:  # fixpoint
            break
    return kb
Мета-політика може бути невеликою мережею (наприклад, MLP над $z_{\text{ctx}}$, кількістю фактів, глибиною), яка навчається через REINFORCE з винагородою за успішне доведення мінус штраф за кроки.

1. От токенов к программам: иерархическая генерация
Вместо предсказания следующего токена $P(w_t | w_{<t})$, OSF строит вывод в виде иерархической программы $\mathcal{P}$, состоящей из уровней абстракции:

Уровень намерений (Intent Level) — символьные цели и подцели (например, "реализовать функцию сложения").

Уровень планов (Plan Level) — последовательность операторов или вызовов функций.

Уровень выражений (Expression Level) — конкретные синтаксические конструкции.

Уровень токенов (Token Level) — последовательность символов/токенов.

Каждый уровень описывается на своем языке: от логических формул до конкретного синтаксиса.

2.5. Мета-контроллер стратегии генерации
Поскольку разные задачи требуют разного баланса между планированием, симуляцией и исправлениями, вводится мета-контроллер уровня генерации, аналогичный EMC для глубины вывода. Он выбирает стратегию синтеза $\sigma \in {\text{Fast}, \text{Careful}, \text{Exploratory}}$, которая определяет:

Глубину планирования.

Количество итераций рефлексии.

Пороги уверенности для принятия результата.

Мета-контроллер обучается через REINFORCE с мета-выигрышем, учитывающим качество, время и вычислительные затраты.

3. Интеграция в OMEN: полная архитектура OSF
OSF интегрируется в OMEN-Scale как расширение декодера, заменяющее простой TokenDecoder.

3.1. Компоненты OSF
Intent Encoder — преобразует $z_{\text{final}}$ из Perceiver в символьную цель.

Symbolic Planner — строит высокоуровневый план с помощью S-Core и нейропоиска.

Hierarchical Decoder — поэтапно уточняет план до синтаксического дерева и токенов.

World Simulator — исполняет код в латентном пространстве (WorldRNN) или символьном интерпретаторе.

Reflection Module — анализирует несоответствия и предлагает правки.

Meta-Controller — управляет процессом, решая, когда остановить уточнение.

3.2. Обучение
OSF обучается end-to-end с использованием комбинации:

Стандартного кросс-энтропийного лосса на уровне токенов (для базового синтаксиса).

Лосса структурного соответствия между планом и кодом.

Лосса симуляции — насколько трасса исполнения соответствует ожиданиям.

REINFORCE для планировщика и мета-контроллера.

Градиенты проходят через все компоненты благодаря использованию дифференцируемых приближений (Gumbel-Softmax для дискретных выборов, Straight-Through для структурных решений).

5. Чому це "надвичайно просунуто"
Универсальность: OSF применима к любой задаче синтеза, где можно определить символьный план (код, текст, речь, движения робота). Переход от плана к реализации осуществляется нейросетевым конкретизатором, который может быть адаптирован под конкретный домен.

Глубокая интеграция с S-Core: Планирование и верификация выполняются на символьном уровне, что гарантирует логическую согласованность и композиционное обобщение.

Внутренняя симуляция: OSF не просто предсказывает следующий токен, а прогоняет код в воображении, чтобы проверить его поведение до того, как выдать пользователю. Это приближает систему к человеческому процессу написания программ.

Самокоррекция: В отличие от большинства генеративных моделей, OSF способна анализировать свои ошибки и исправлять их, используя цикл рефлексии. Это критически важно для надежности.

Мета-оптимизация: OSF учится как генерировать в зависимости от сложности задачи, балансируя между скоростью и тщательностью. Это делает систему адаптивной и эффективной.

Передбачити нові факти, які мали б випливати з правил, але не були явно виявлені увагою.

6. Взаємний вплив компонентів (цикл)
Трансформер (TokenEncoder/NET) генерує увагу → Saliency Trace → факти в WM.

S-Core намагається пояснити ці факти правилами, виводить нові факти, оновлює $Q_{\text{S-Core}}$.

Role Classifier отримує градієнт від $\mathcal{L}_{\text{role}}$, вчиться передбачати ролі, які відповідають правилам.

Трансформер через $\mathcal{L}_{\text{struct}}$ отримує градієнт до своїх представлень, вчиться генерувати такі $z$, які добре узгоджуються з символьним графом.

Абдукція створює нові правила, якщо увага систематично показує зв'язки, які не пояснюються існуючими правилами.

VeM та Epistemic Tracker оцінюють корисність правил і видаляють погані.

Цей цикл забезпечує постійне самонавчання без необхідності зовнішнього розміченого корпусу для семантичних ролей — система сама відкриває лінгвістичні структури через узгодження нейронної статистики та символьного MDL.

7. Висновок
Saliency Trace в ідеалі — це міст між неперервною увагою трансформера та дискретною логікою S-Core. Математично це реалізується через:

Витяг графа уваги з трансформера.

Перетворення його на Horn-факти за допомогою навченого Role Classifier.

Оцінку узгодженості з правилами LTM та абдукцію нових правил.

Двонаправлений градієнтний потік: від S-Core до Role Classifier ($\mathcal{L}{\text{role}}$) та до трансформера ($\mathcal{L}{\text{struct}}$).

Природа рольових міток: апріорне визначення vs динамічне навчання
Словник семантичних ролей $\mathcal{R} = {\text{agent}, \text{patient}, \text{action}, \text{modifier}, \text{coref}, \dots}$ є апріорно заданим — він визначається розробником на етапі проєктування системи як частина онтології предметної області (лінгвістики). Це аналогічно до того, як предикати в S-Core (assign, call, type_of) закладені заздалегідь. Самі мітки не вивчаються з нуля — вони є фіксованими категоріями, які система використовує для інтерпретації світу.

Однак приписування конкретних міток конкретним токенам у кожному реченні не задається заздалегідь. Role Classifier $R_\phi$ — це нейромережа, яка навчається передбачати розподіл $P_\phi(r \mid x_i)$ без зовнішнього розміченого корпусу. Вона отримує навчальний сигнал виключно через узгодження з S-Core, тобто через $\mathcal{L}{\text{role}}$ та $\mathcal{L}{\text{struct}}$.

Таким чином, система самостійно відкриває, які патерни уваги відповідають, скажімо, відношенню "агент-дія", шляхом мінімізації розбіжності з логічними очікуваннями S-Core. Це є формою self-supervised learning, де наглядачем виступає символьний рівень.

Математична формалізація навчання Role Classifier без розмітки
Нехай $\mathcal{D}$ — немаркований корпус текстів. Для кожного речення $X \in \mathcal{D}$:

Трансформер генерує приховані стани $h_i$ для кожного токена $x_i$.

Role Classifier обчислює $P_\phi(r \mid x_i) = \text{softmax}(W_R h_i + b_R)$.

S-Core отримує Saliency Trace $\mathcal{F}_{\text{sal}}$ (факти про зв'язки між токенами) і застосовує правила з LTM.

На основі активованих правил S-Core будує цільовий розподіл $Q_{\text{S-Core}}(r \mid x_i)$. Наприклад, якщо правило $R$ має вигляд agent(?Act, ?Ag) :- action(?Act), subject(?Act, ?Ag), і в WM є факти action(tok5), subject(tok5, tok2), то S-Core очікує, що роль agent має бути призначена токену tok2 з високою ймовірністю, а роль action — токену tok5. Формально, $Q_{\text{S-Core}}$ — це дельта-розподіл або згладжений розподіл на основі виведених фактів.

Обчислюється $\mathcal{L}{\text{role}} = \frac{1}{T} \sum{i} \text{KL}(P_\phi(\cdot \mid x_i) ,|, Q_{\text{S-Core}}(\cdot \mid x_i))$.

Градієнт цього лосу проходить через параметри $\phi = {W_R, b_R}$, оновлюючи їх так, щоб передбачення Role Classifier наближалися до очікувань S-Core.

Отже, рольові мітки як категорії визначені людиною, але їх конкретне застосування до даних вивчається системою автономно через нейро-символьний зворотний зв'язок. Це поєднує переваги структурованої символіки (інтерпретованість, композиційність) із гнучкістю нейромережевого навчання (адаптація до реальних даних).
